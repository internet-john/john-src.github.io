Why React?

An odd question to ask in 2020, as the library has matured and become the choice of individuals, startups, and enterprise alike.

Rly tho. Why react? What makes it so special, and how did we get here?

The intent of this write up is to give the reader insight into React design principles and implementation, and what makes it so special.

A shifted paradigm

A large part of traditional web development followed the MVC paradigm. Manage models, views, controllers. Manage the instances of these controllers, their state, their properties. Update the DOM accordingly. Rinse and repeat. As applications scaled and grew in features and complexity, it became clear that MVC design patterns and the push approach burdered developers with additional cognitive overload.

If only there was a library to build interfaces that addressed key pain points of web development...

Boy, do I have news for 2013 you!

The Tao of React

As discussed above, their was a number of issues with traditional web development. Let’s discuss these one by one and see how React’s design addresses this.

  • DOM manipulation
    • In React, your UI components, whether they be custom () or built-in such as div, it’s all good. Your renders will plain JS objects, which properties that describe their type, properties, and children. These objects are not instances. In React, they are all elements.

    An element is a plain object describing a component instance or DOM node and its desired properties.

  • Instance management

    • As stated above, your UI components are all elements in React land. When a component updates, the state changes but the instance remains the same. Nice.

  • UI updates

    • This is handled by React reconciliation. The algo relies on two key heuristics: 1) Elements of different types produce different trees 2) List items contain a key property that unique, stable, and predictable as the component renders, re-renders, and re-renders again.

    When a component is updated, it generates a new tree of JS objects. This tree is diffed against the previous one. If the root elements are the same type, it moves on to the attributes and styles, updating as necessary. It will then repeat this for each child node until the tree has been updated where needed. If the root elements are of different type, it will tear down the tree and rebuild anew. This new, updated tree is then passed to the renderer depending on environment. (ReactDOM for web, React-Native for mobile) As we can see, React has taken a lot of the work devs would previously managed.

    It is a key goal for React that the amount of the user code that executes before yielding back into React is minimal.

TL;DR

In summary, React is simply a library fine-tuned for building UI’s. Used properly, it removes a lot of headaches web developers would face in every day development.
